## 1. URL 与 locale 结构

Hydrogen 通过 URL 前缀区分 locale：

```
/products/t-shirt           → 默认（EN-US）
/en-US/products/t-shirt     → English (United States)
/zh-CN/products/t-shirt     → 中文（中国）
/ja-JP/products/t-shirt     → 日本語（日本）
/fr-FR/products/t-shirt     → Français (France)
```

### 1.1 解析逻辑

`app/lib/i18n/i18n.ts`：

```tsx
export interface I18nLocale extends I18nBase {
  pathPrefix: string // '' 或 '/en-US'
}

export function getLocaleFromRequest(request: Request): I18nLocale {
  const url = new URL(request.url)
  const firstPathPart = url.pathname.split('/')[1]?.toUpperCase() ?? ''

  let pathPrefix = ''
  let [language, country] = ['EN', 'US'] // 默认值

  // 匹配 XX-YY 格式
  if (/^[A-Z]{2}-[A-Z]{2}$/i.test(firstPathPart)) {
    pathPrefix = `/${firstPathPart}`
    ;[language, country] = firstPathPart.split('-')
  }

  return { language, country, pathPrefix }
}
```

### 1.2 locale 生效的三个位置

**① GraphQL 查询自动注入 `@inContext`**

```graphql
query Product($country: CountryCode, $language: LanguageCode)
  @inContext(country: $country, language: $language) {
  product(handle: "t-shirt") {
    title          # 返回对应语言的标题
    priceRange {
      minVariantPrice {
        amount       # 返回对应国家的货币和价格
        currencyCode
      }
    }
  }
}
```

`$country` 和 `$language` 由 `context.storefront.i18n` 自动传入，**不需要手动传递**。

**② locale 路由校验** — `app/routes/($locale).tsx`：

```tsx
export async function loader({ params, context }) {
  const { language, country } = context.storefront.i18n
  if (params.locale
    && params.locale.toLowerCase() !== `${language}-${country}`.toLowerCase()) {
    throw new Response(null, { status: 404 })
  }
  return null
}
```

**③ 内部链接前缀** — 内部跳转需要带 locale 前缀：

```tsx
function getVariantUrl({ handle, pathname }) {
  const match = /(\/[a-zA-Z]{2}-[a-zA-Z]{2}\/)/.exec(pathname)
  return match
    ? `${match[0]}products/${handle}`
    : `/products/${handle}`
}
```

### 1.3 当前实现的扩展点

目前是最简模式（URL 前缀）。可改造为：

- **Cookie-based**：从 Cookie 读取用户偏好
- **Header-based**：从 `Accept-Language` 推断
- **Subdomain-based**：`us.myshop.com` / `jp.myshop.com`
- **混合模式**：URL 前缀 + Cookie 记住偏好

只需改 `getLocaleFromRequest()`。

## 2. 翻译层级地图：哪些字段会被 Shopify 翻译

**这张表是避坑的核心**。开发时必须清楚：在 `@inContext(language: JA)` 下，哪些字段是日语、哪些还是英语。

| 字段 | 是否受 `@inContext(language)` 影响 | 会被 Translate & Adapt 翻译 | 代码能依赖其稳定性吗？ |
|------|:--:|:--:|:--:|
| `product.id` | ❌ | ❌ | ✅ 全局稳定 |
| `product.handle` | ⚠️ 可能被翻译 | ✅（URL handle 字段） | ⚠️ 需要 `redirectIfHandleIsLocalized` 处理 |
| `product.title` | ✅ 返回翻译 | ✅ | ❌ 显示用，别做匹配 |
| `product.description` | ✅ | ✅ | ❌ |
| `product.options[i].name` | ✅（**！坑**） | ✅ Option name 翻译项 | ❌ |
| `product.options[i].optionValues[j].name` | ✅（**！坑**） | ✅ Option values 翻译项 | ❌ |
| `variant.id` | ❌ | ❌ | ✅ 全局稳定 |
| `variant.selectedOptions[i].name` | ✅ | ✅ | ❌ |
| `variant.selectedOptions[i].value` | ✅ | ✅ | ❌ |
| `variant.price.amount` | ❌ 受 `country` 影响 | ❌ | ⚠️ 每 market 不同 |
| `variant.sku` | ❌ | ❌ | ✅ |

**结论**：代码做任何 variant / option 的匹配或判断，**只能**依赖 `id`、`sku`、顺序位置 —— **不要**依赖 name/value 文本。

## 3. Variant 匹配：三种方案对比

这是踩最多坑的地方。多语言站点上，用户选一套选项组合（颜色 / 版本 / 订阅方案）→ 要找到对应的 variant。

### 3.1 ❌ 文本匹配（历史写法，翻译后必崩）

```ts
// 曾经的写法，不要再用
variants.find(v =>
  v.selectedOptions.some(o => o.name === 'Colour' && o.value === 'Satin Gunmetal')
  && v.selectedOptions.some(o => o.name === 'Edition' && o.value === 'Standard'),
)
```

**为什么崩**：`@inContext(language: JA)` 下 Shopify 返回的 `name` 变成 `色`、`value` 变成 `標準` → 英文硬编码全部对不上 → `find` 返回 `undefined` → 按钮被禁用。

**真实案例**：2026-04 市场部在 Translate & Adapt 里对 Flowtica Scribe 的 Option name/values 点了 Auto-translate，导致日语站所有「立即购买」按钮灰掉。排查链路见 Git 历史。

### 3.2 ✅ 位置索引匹配（本项目当前方案）

**原理**：`product.options[0]` 永远是第 1 个 option（即使名字翻译为「色」），`optionValues[0]` 永远是第 1 个值。UI 自己维护选项顺序，用 index 拼 key。

**loader**（[app/routes/products/$handle.tsx](../app/routes/products/$handle.tsx)）：

```ts
const variantLookup: Record<string, string> = {}
const productOptionValues = (product.options ?? []).map(
  opt => (opt.optionValues ?? []).map(ov => ov.name),
)
for (const v of product.variants?.nodes ?? []) {
  const positions = (v.selectedOptions ?? []).map(
    (selOpt, i) => productOptionValues[i]?.indexOf(selOpt.value) ?? -1,
  )
  if (positions.every(p => p >= 0)) {
    variantLookup[positions.join('|')] = v.id // "0|1|2" → variant.id
  }
}
return { product, variantLookup }
```

**组件**（[app/components/buy-now/BuyNowProductPurchase.tsx](../app/components/buy-now/BuyNowProductPurchase.tsx)）：

```ts
const selectedVariant = useMemo(() => {
  const colorPos = COLOR_OPTIONS.findIndex(o => o.id === selectedColorId)
  const editionPos = SET_OPTIONS.findIndex(o => o.id === selectedSetId)
  const planPos = PLAN_OPTIONS.findIndex(o => o.id === selectedPlanId)
  const variantId = variantLookup[`${colorPos}|${editionPos}|${planPos}`]
  return variantId ? variants.find(v => v.id === variantId) : undefined
}, [variants, variantLookup, selectedColorId, selectedSetId, selectedPlanId])
```

**优点**：
- 一次 GraphQL 搞定，不需要额外查英文数据
- 翻译怎么改都不崩，UI 文案完全解耦
- 运营随便翻

**前提**：UI `COLOR_OPTIONS` / `SET_OPTIONS` / `PLAN_OPTIONS` 的顺序必须和 Shopify admin 里 Option Values 的顺序一致。换 Shopify 顺序时要同步改 UI 常量。

### 3.3 ✅ 官方方案：`selectedOrFirstAvailableVariant`（URL 驱动）

**原理**：不在前端做 variant 匹配。把 URL 的 query params 作为 `selectedOptions` 传给 Shopify，由 Shopify **服务端**解析出 variant。Shopify 自己知道翻译映射。

```graphql
query Product($selectedOptions: [SelectedOptionInput!]!, ...) {
  product(handle: $handle) {
    selectedOrFirstAvailableVariant(
      selectedOptions: $selectedOptions
      ignoreUnknownOptions: true    # 未知 key 不报错
      caseInsensitiveMatch: true    # 大小写无关
    ) { ...ProductVariant }
  }
}
```

loader 里：

```ts
storefront.query(PRODUCT_QUERY, {
  variables: {
    handle,
    selectedOptions: getSelectedProductOptions(request), // 从 URL 原样提取
  },
})
```

组件里直接读 `product.selectedOrFirstAvailableVariant`，配合 `useOptimisticVariant` + `useSelectedOptionInUrlParam` + `getProductOptions`。

**参考**：官方 skeleton 模板 [templates/skeleton/app/routes/products.$handle.tsx](https://github.com/Shopify/hydrogen/blob/main/templates/skeleton/app/routes/products.%24handle.tsx)。

**什么时候用**：新页面用原生 Shopify 选择器 UI（没有自定义的 UI ID / 翻译 key 体系）。

**什么时候别用**：Flowtica Scribe 这种重度自定义 UI（颜色色板、订阅方案视觉卡片、附加商品选择等 —— 想保留自有 label/i18n key 体系），按 3.2 位置匹配更合适。

## 4. URL query 参数的三类写入源

多语言站点上 URL 会出现不同来源写的 query params。调试前先识别来源。

| 来源 | 写入的 key 是什么语言 | 备注 |
|------|---------|------|
| **自定义组件**（项目内） | 英文（如 `color` / `edition` / `plan`） | 由我们自己控制 |
| **`useSelectedOptionInUrlParam`**（Hydrogen hook） | **当前 locale**（`color` 可能变 `色`） | hook 写的是 `selectedVariant.selectedOptions.name`，受 `@inContext(language)` 影响 |
| **Shopify 默认 variant selector** / 站外复制的链接 | 当前 locale | Shopify 官方 picker 默认用翻译后 name 当 URL key |

**后果**：日语页面 URL 可能同时出现 `color=xxx` 和 `色=xxx`，一个英文 key 一个日文 key。

**解决**：
- 自定义组件内部只读自己写入的 key（英文 `color` / `edition` / `plan`）
- 清洗策略：在 `setSearchParams` 之前先 `delete` 所有可能的 locale 化版本
- 或彻底不用自定义 URL key，改走方案 3.3

## 5. 本仓库踩过的真实坑（持续更新）

### 5.1 `storefront.query` 传 `language` override **不生效**

**现象**：想在 loader 里额外多查一次英文 variants 做跨语言匹配表：

```ts
storefront.query(QUERY, {
  variables: {
    handle,
    country: context.storefront.i18n.country,
    language: 'EN',  // ❌ 这个 override 不生效
  },
})
```

**结果**：Shopify 依然按 `storefront.i18n.language`（即当前 locale）返回 —— `selectedOptions` 里值还是日语。

**原因**：当前版本的 Hydrogen `storefront.query` 把 `context.storefront.i18n` 硬合并进 variables，用户传的 `language` 会被覆盖。

**规避**：
- 方案 A（推荐）：改用 3.2 位置索引，不依赖英文数据
- 方案 B：绕过 `storefront.query`，直接 `fetch` Storefront API 端点（需要手动拼 token、headers）

### 5.2 Translate & Adapt 对 Product options 点 Auto-translate 必炸

**现象**：商品页「立即购买」按钮全灰。

**原因**：Auto-translate 把 `Option name`（如 Colour→色）和 `Option values`（如 Standard→標準）都翻译了 → 代码里任何基于这些文本的匹配全失败。

**规避**：
- 翻译 Product 时只翻 Title / Description / Meta / Policies
- **Option name / Option values 一律不翻**
- 或者使用方案 3.2 位置匹配，翻不翻都不崩

### 5.3 Market 拆分后 Checkout 语言 fallback 到英文

**现象**：日文站点点 Continue to Checkout → 跳转到英文结账页。

**原因**：Checkout 页的语言**不是由 URL locale 或 `@inContext(language)` 决定的**，而是由 cart `buyerIdentity.countryCode` 所属的 Market 里的 Language 配置决定。

**2026-03 真实案例**：运营把 `Asia-Japan` market 拆成独立的 `Japan` 和 `Singapore`。新 Japan market 是**全新创建的**，没有绑定日语 → JP 订单的 checkout 全部 fallback 到 primary language（英文）。

**排查链路**：

```
[Settings → Markets → Japan → Languages] 是否有 Japanese 且 Published？
        ↓ 没有
点 Add language → Japanese → Publish
        ↓ 有
[Settings → Languages → Japanese → Markets] 有没有新 Japan？
        ↓ 没有
Assign 到新 Japan market
        ↓ 都对
清 cart cookie（旧 cart 仍指向已删除的 Asia-Japan market）
        ↓ 仍错
GraphiQL 复现：mutation @inContext(country: JP, language: JA) { cartCreate(...) { cart { checkoutUrl } } }
浏览器打开 checkoutUrl → 在 Shopify 侧复现 → 走 Shopify Support
```

### 5.4 Cart mutation 返回数据不遵守 `@inContext(language)`

**现象**：cart 里商品标题是英文，但 checkout 页是日文。

**原因**：已知 Shopify bug —— cart mutation 返回的 `cart.lines[].merchandise.product.title` 不走当前 `@inContext` 的 language 参数。

**规避**：
- 购物车侧的商品标题从自己的 i18n 字典拿，不要直接显示 cart 返回的 title
- `checkoutUrl` 本身的语言绑定**不受此 bug 影响**，保持正常

## 6. 新增语言的完整 Checklist

每次新增语言（如简体中文、韩语）时按顺序做完：

### 运营侧

- [ ] Shopify 后台 `Settings → Languages → Add language` → 选语言 → **Publish**
- [ ] `Settings → Languages → <新语言> → Manage domains` 分配域名
- [ ] `Settings → Markets → <对应国家/区域 market> → Languages` 绑定新语言
- [ ] 如果目标国家没有单独 market，创建新 market 并配置 Currency / Languages / Domain
- [ ] Translate & Adapt 里填翻译：**只翻 Title / Description / Meta / Navigation / Pages / Policies / Email notifications**
- [ ] **不翻**：Product options / Option values / URL handle

### 代码侧

- [ ] [app/lib/i18n/i18n.ts](../app/lib/i18n/i18n.ts) 的 `SUPPORTED_LOCALES` 追加一项：
      ```ts
      { language: 'ZH', country: 'CN', pathPrefix: '/ZH-CN', label: '简体中文' }
      ```
- [ ] 如果某组件有 `isJa` 这种硬编码 locale 分支（如 [BuyNowProductPurchase](../app/components/buy-now/BuyNowProductPurchase.tsx) 的 gallery 图），同步新增 `isZh` 分支
- [ ] i18n 字典文件 `app/locales/zh-CN/*.json` 添加（参考 `en-US/` 结构）
- [ ] locale switcher 里加入新语言

### 验证（每种语言都要跑）

- [ ] GraphiQL `query @inContext(country: CN, language: ZH) { product(handle: "...") { title } }` 返回中文
- [ ] 前端 `/ZH-CN/products/<handle>` 商品页中文显示
- [ ] 切换选项，「立即购买」可点（位置匹配不受翻译影响）
- [ ] 加购 → cart 页价格正确（本 market 的货币）
- [ ] 点 Continue to Checkout → 跳转后 checkout 页本语言显示（**不是** fallback 英文）
- [ ] 下一个真实测试订单，订单邮件是本语言

## 7. 本地调试单一 locale 的快捷方式

开发时要快速切换 locale 模拟不同市场用户，几种方式：

```bash
# 方式 1：直接改 URL
http://localhost:3000/JA-JP/products/flowtica-scribe
http://localhost:3000/ZH-CN/products/flowtica-scribe

# 方式 2：清 cart cookie + 刷新
# 浏览器 DevTools → Application → Cookies → 删除 cart 相关项
```

如果按钮灰了 / variant 不对 / 价格不对，按下面顺序排查：

1. **查数据源**：loader 里打印 `context.storefront.i18n`、`product.options[0].optionValues` → 确认 Shopify 返回了什么
2. **查匹配**：组件里打印 `variantLookup` + 拼出来的 key → 是否命中
3. **查 URL**：URL query 键是英文还是当前 locale？混合？
4. **查 cart cookie**：老 cart 绑着旧 market？清了再试
5. **查 market**：Shopify 后台 `Markets → <目标国> → Languages` 有没有目标语言
